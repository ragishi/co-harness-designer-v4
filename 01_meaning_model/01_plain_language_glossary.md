# 01_plain_language_glossary.md — 初めて読む人のための用語集

## 役割

V4 の専門用語を、コードもエンジニアリングも知らない人でも分かる言葉で説明する「言い換え集」です。
難しい英語や略語のそばに「= やさしい言い換え」を添えることで、初めて読む人が迷わないようにします。

## 持つもの

- 12 用語のやさしい言い換え
- 各用語の日常的な比喩
- V4 での具体例

## 持たないもの

- 詳細定義（→ `10_entities.md` / contracts / archetypes）
- 個別 metric 計算式
- 専門的な実装仕様

## この glossary の使い方

1. 分からない用語が出てきたら、番号順に関係なく 1 用語ずつ読んでください。
2. 「もっと詳しく知りたい」と思ったら、各節末の「正本場所への pointer」へ進んでください。
3. 初回は最初から通して読むと V4 の全体像が掴めます。

## 1. scaffold = 雛形セット

### やさしく言うと

新しい Repo（= 作業部屋）を作るときに最初から用意されている、ファイルやフォルダのひな型セットです。
一から全部手で作らなくても、必要なものが最初から揃っています。

### もう少し詳しく

IKEA の組み立てキットに似ています。
箱を開けると、板・ネジ・説明書がすでに揃っている状態です。
自分で木を切ったり金属を溶接する必要はなく、説明書の通りに組み立てるだけで部屋の棚が完成します。
scaffold も同じで、「目的に合った Repo の最低限の構成」があらかじめセットになっています。

### V4 での具体例

- `06_build/` フォルダに archetype ごとの scaffold 定義があります。
- 例：note 制作用の scaffold には `workspace/drafts/` や `outputs/published/` が含まれます。

### 正本場所への pointer

→ `../06_build/` および `10_entities.md` §1

## 2. exporter = UI 表示用に変換する係

### やさしく言うと

Markdown（= テキストで書かれた正本）の内容を、CompanyOS（= 管理画面）で見やすい形に変換する担当です。
正本は変えず、表示用のコピーだけを作ります。

### もう少し詳しく

料理本の翻訳係に似ています。
フランス語で書かれたレシピ（= 正本）があるとします。
翻訳係（= exporter）は、そのレシピを日本語に訳して別冊子（= 表示用コピー）に印刷します。
フランス語の原本は変わりません。翻訳版が変わっても、原本を書き直したことにはなりません。

### V4 での具体例

- `06_build/` にある exporter 定義が、Markdown を読んで `.companyos/` 配下に JSON を生成します。

### 正本場所への pointer

→ `../02_contracts/40_surface_contract.md` および `10_entities.md` §6

---

## 3. generated = 正本から作った表示用コピー

### やさしく言うと

Markdown の正本をもとに自動で作られた「表示専用のファイル」です。
手で直接編集してはいけません。正本を直すと、generated も自動で更新されます。

### もう少し詳しく

学校の成績表の「印刷版」に似ています。
先生の手元にある成績記録（= 正本）を元に、プリンターが印刷した用紙（= generated）が配られます。
配られた用紙に書き込んでも成績は変わりません。変えたければ、先生の元帳を直す必要があります。

### V4 での具体例

- `.companyos/` 配下の JSON ファイルはほぼすべて generated です。
- `source_pin`（→ 6 番）でどの Markdown から作ったかが分かります。

### 正本場所への pointer

→ `../02_contracts/40_surface_contract.md` および `../00_foundation/02_ssot_authority_map.md`

## 4. surface = UI への見せ方

### やさしく言うと

Repo の中身を CompanyOS の画面にどう表示するかを決めた「見せ方のルール」です。
同じ情報でも、surface を変えると画面上の見え方が変わります。

### もう少し詳しく

展覧会の展示方法に似ています。
同じ絵画（= Repo の情報）でも、額縁の大きさ・照明・壁の色（= surface の設定）によって
観客に与える印象が変わります。絵画自体は変わっていません。

### V4 での具体例

- `02_contracts/40_surface_contract.md` に surface のルールが書かれています。
- note archetype の surface は「記事タイトル・公開日・本文プレビュー」などが含まれます。

### 正本場所への pointer

→ `../02_contracts/40_surface_contract.md` および `10_entities.md` §6

---

## 5. lens = 目的別の見え方

### やさしく言うと

同じ Repo の情報を、目的に合わせて「どこをクローズアップするか」を変えた「フィルター」です。
同じデータでも、lens を切り替えると別の角度から見えます。

### もう少し詳しく

スマートフォンのカメラフィルターに似ています。
撮っている景色は同じでも、「ポートレートモード」「広角モード」「ナイトモード」にすると見え方が変わります。
lens も同じで、「進捗ビュー」「カレンダービュー」「担当者ビュー」などで同じ情報を切り替えます。

### V4 での具体例

- `11_connect/10_companyos_connection.md` に Workflow Lens / Member Work Lens / Repo Card の説明があります。
- 同じ進捗情報も、Workflow Lens では「kanban 風」、Member Work Lens では「人ごとの TODO」に見え方が変わります。

### 正本場所への pointer

→ `../11_connect/10_companyos_connection.md`（Workflow Lens / Member Work Lens / Repo Card の説明）および `10_entities.md` §7（Lens entity の定義）

## 6. archetype = 目的別 Repo の型

### やさしく言うと

「note 制作用の Repo はこういう構成にする」「client 案件用はこういう構成にする」という、
目的ごとに決められた Repo の「モデルパターン」です。

### もう少し詳しく

家の「間取りの型」に似ています。
不動産では「1LDK」「2LDK」「一戸建て」などの型があります。
住む人の目的（一人暮らし / 家族 / テレワーク重視）に合わせて、最初から最適な間取りを選べます。
archetype も同じで、Repo の目的に合ったフォルダ構成・ファイル・ルールがあらかじめセットになっています。

### V4 での具体例

- `04_archetypes/repo/` には**現在の MVP では note / client の 2 archetype** が定義されています。残り 6 種（project_progress / sns_operations / youtube_ops / workflow_ops / prompt / design_system）は後段フェーズで追加されます。
- archetype を選ぶと、対応する scaffold（→ 1 番）が決まります。

### 正本場所への pointer

→ `../04_archetypes/` および `10_entities.md` §15

## 7. source_pin = 元情報の記録

### やさしく言うと

generated ファイル（→ 3 番）が「どの Markdown から作られたか」を記録した印（しるし）です。
「このコピーの元ネタはどこ？」に答えてくれます。

### もう少し詳しく

図書館の「出典ラベル」に似ています。
コピー機でコピーした資料（= generated）には「出典：〇〇図書館 〇〇棚 〇〇本 p.xx」というラベルが貼られます。
そのラベル（= source_pin）を見れば、元の本（= 正本 Markdown）がどこにあるかすぐ分かります。

### V4 での具体例

- `.companyos/` 配下の JSON に `source_pin: "workspace/drafts/article-01.md"` のように書かれます。

### 正本場所への pointer

→ `../02_contracts/40_surface_contract.md`

## 8. read_set = AI が読むべきファイル一覧

### やさしく言うと

ある作業を始める前に AI が読んでおくべきファイルの「おすすめ参考書リスト」です。
作業の種類によって、読む必要があるファイルが変わります。

### もう少し詳しく

試験勉強の「出題範囲リスト」に似ています。
「数学の試験では教科書 3 章・4 章・問題集 p.50–60 を読め」と指定されたリストがあれば、
全部を最初から読まなくても必要な部分だけ効率よく準備できます。
read_set も同じで、「この作業なら、まずこのファイルを読め」と事前に指定されています。

### V4 での具体例

- `05_read_sets/` フォルダに作業別の read_set が入っています。
- 例：「新規 archetype を追加する」作業の read_set には `04_archetypes/` と `02_contracts/` が含まれます。

### 正本場所への pointer

→ `../05_read_sets/` および `10_entities.md` §13

## 9. MNP = UI 状態の中間メモ

### やさしく言うと

CompanyOS の画面（UI）の状態を、AI が読み書きしやすいテキスト形式に変えた「下書きメモ」です。
UI を直接操作せずに、テキストで UI の内容を確認・更新するための中間形式です。

### もう少し詳しく

授業ノートの「清書前の走り書き」に似ています。
授業中に黒板を見ながらノートに走り書きし（= MNP）、後で清書して清書版（= UI 表示）に反映します。
走り書きの段階で内容を確認・修正できるので、清書版を直接書き直す手間が省けます。

### V4 での具体例

- `.companyos/mnp/` 配下に MNP ファイルが置かれます。
- exporter（→ 2 番）が MNP を読んで、UI 表示用の JSON に変換します。

### 正本場所への pointer

→ `../02_contracts/40_surface_contract.md` および `10_entities.md` §16

## 10. SSOT = 情報の本籍地

### やさしく言うと

「この情報の正しい版はここにある」と決められた唯一の場所です。
同じ情報が複数の場所にあるとき、どれが本物か迷わないように「ここが本籍地」と決めておきます。

### もう少し詳しく

住民票の「本籍地」に似ています。
引っ越しして住所が変わっても、本籍地は変わりません。
「この人の戸籍情報はここにある」という唯一の場所が決まっているので、
どこを見れば本当の情報が分かるか迷いません。
SSOT も同じで、「この情報は Markdown ファイルが正本で、JSON は表示用コピー」と決まっています。

### V4 での具体例

- V4 全体のルールとして「Markdown が SSOT、JSON は generated」です。
- `00_foundation/02_ssot_authority_map.md` に何が SSOT かの地図が書かれています。

### 正本場所への pointer

→ `../00_foundation/02_ssot_authority_map.md`

## 11. contract = 連携の約束

### やさしく言うと

V4 の各パーツ（Repo・AI・UI・exporter など）が「どんな形式でやり取りするか」を決めた「約束書」です。
contract があるおかげで、パーツ同士が安心して連携できます。

### もう少し詳しく

引っ越し業者との「見積書兼約束書」に似ています。
「何を・どこから・どこへ・いつ・どんな箱に入れて運ぶか」が書かれた書面があれば、
業者と依頼主が別々に動いても、当日に「聞いていない」と揉めません。
contract も同じで、「このファイルはこのフォーマットで渡す」「この項目は必須」などが決まっています。

### V4 での具体例

- `02_contracts/` フォルダに各種 contract が入っています。
- 例：`40_surface_contract.md` は「Markdown → generated JSON」の変換ルールを定めます。

### 正本場所への pointer

→ `../02_contracts/`

## 12. optional module = 目的に応じて足す部品

### やさしく見ると

Repo の基本セット（scaffold）に、必要なときだけ追加できる「追加パーツ」です。
最初から全部入れる必要はなく、目的が増えたら後から足します。

### もう少し詳しく

スマートフォンのアプリインストールに似ています。
スマートフォンを買ったとき、最初から全アプリが入っているわけではありません。
「カメラアプリは最初から必要。地図アプリは旅行前に入れる。料理アプリは後から興味が出たら入れる」
という感じで、必要なときに必要なものだけを追加します。
optional module も同じで、Repo の目的が広がったときに後から追加します。

### V4 での具体例

- `40_extensions/` フォルダに optional module が入っています。
- 例：note archetype に「SNS 投稿連携モジュール」を後から足すことができます。

### 正本場所への pointer

→ `../40_extensions/` および `10_entities.md` §14
